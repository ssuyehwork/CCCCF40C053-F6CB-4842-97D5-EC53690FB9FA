## 🗣️ 零号定律：绝对沟通原则 (Communication Core)
1. **【强制纯中文】**：任何情况下，只准使用中文与用户对话，绝对不允许使用英语或中英夹杂！

3. **【专注当前，禁止回头】**：顺着用户的思路走，已经解决或修改过的问题，**绝对不可再反反复复去纠结或重构**。你的视野只能聚焦于“当前最新的任务”，杜绝任何“状态串线”或“脱节”行为。

## 🛑 第一定律：强制“握手-审批”状态机 (防先斩后奏)

- **[状态 1: 分析与反问]**：只能输出纯文本计划。必须排查需求中的盲区（异常处理、UI细节等）。如果有不懂的，**绝对不准脑补，必须向用户提问并提供选项！** 每次必须确认你的理解是否与用户达成共识。

  `🛑 [系统锁定]：我已准备好执行计划，确认共识。请回复【批准修改】，我才会开始触碰代码。`

## ⚙️ 第二定律：C++/Qt 编译与架构防爆红线 (防编译失败)
每一次代码输出，必须保证一次性编译通过并追求极限性能：
1. **【强制包含头文件 #include】**：引入任何新的 Qt 类或 C++ 标准库时，**强制第一优先级去文件顶部检查并添加 `#include <XXX>`！** 漏加头文件是不可饶恕的死罪。
2. **【头文件与源文件强制同步】**：在 `.cpp` 里实现新函数、信号或槽函数，**必须同时去对应的 `.h` 头文件中声明它！** 包含必要的前置声明和 `Q_OBJECT` 宏。
3. **【移除代码的上下文安全检查】**：在移除任何旧代码前，必须进行全局上下文推演，确认移除该代码不会导致其他模块报错或逻辑链断裂。
4. **【极限性能与架构审查】**：必须充分发挥 C++ 的极限性能。在编写复杂逻辑时，主动分析可能的架构瓶颈并给出优化建议；优先使用智能指针 (`std::unique_ptr` 等) 防止内存泄漏。

## ☢️ 第三定律：Qt 运行时生命红线 (防崩溃闪退)
1. **【严禁后台线程更新 UI】**：如果使用了 `QThreadPool` 或其他后台线程进行耗时操作，**绝对禁止**在后台线程里直接调用 UI 组件的方法。必须使用 `signals/slots` 或 `QMetaObject::invokeMethod` 抛回主线程执行！
2. **【Lambda 捕获陷阱】**：异步操作中，严禁盲目使用 `[=]` 或 `[&]` 捕获 `this` 指针，防止对象销毁导致的悬垂指针崩溃。

## 📐 第四定律：业务规范与 UI 绝对禁区 (DO NOT TOUCH)
以下项目独有规范在底层被物理封锁，如果你试图绕过，将立即被终止运行：
1. **【唯一提示组件】**：**严格遵守项目中关于“禁止使用原生 Windows Tip 样式”的规定！** 任何待办日历、按钮或组件的提示逻辑，必须且只能迁移调用 `ToolTipOverlay.h` 中的实现。
2. **【QuickPreview 标题栏规范】**：预览窗口的标题栏按钮必须遵循从右到左的固定顺序：**关闭 → 最大化 → 最小化 → 置顶 → 编辑**。不准擅自调整！
3. **【UI 样式冻结】**：绝不允许私自修改任何未要求的 `padding`, `margin` 相关的 QSS 样式代码。
4. **【禁止调用底层键盘事件】**：严禁直接在业务模块使用原生拦截注册全局热键。必须调用项目中封装好的热键工具函数。

## 🏗️ 第五定律：代码质量与维护追踪 (DRY & 溯源)
1. **【强制行为意图注释】**：**每次修改代码，必须在修改处加上清晰的中文注释，说明用户的要求/意图**（例如：`// 2026-03-xx 按照用户要求，修改按钮顺序为关闭在最右侧`）。防止后续被其他 AI 胡乱篡改，除非用户明确要求删除。
2. **【全局查重】**：新增工具函数前，必须先搜索全局项目复用现有代码，杜绝复制粘贴的冗余。
3. **【消灭魔法数字】**：业务逻辑中严禁出现裸露数字和硬编码中文字符串，必须提取为 `const` 常量。
4. **【防御性编程】**：严禁深层嵌套 `if-else`，必须优先检查边界条件并在函数开头 `return`。
